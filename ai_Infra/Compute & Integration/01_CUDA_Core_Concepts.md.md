## 01 - CUDA 核心执行模型与内存架构

在编写高性能 CUDA 算子之前，必须深刻理解 GPU 的软硬件映射机制、执行模型和内存层级。

### 1. 宏观执行流程与软硬件映射 (Grid / Block / SM)

GPU 的强大在于其庞大的并行规模和零开销的上下文切换机制。理解 GPU 编程，首先要建立软件逻辑与物理硬件的映射关系。

- **软件逻辑层 (程序员视角)：** * **Grid (网格)：** 代表整个计算任务。
    
    - **Block (线程块)：** 为了“组队协作”，我们将海量线程划分为一个个 Block。**同一个 Block 内的线程享有两大特权：可以通过 Shared Memory 高速通信，也可以进行强制同步 (`__syncthreads()`)。** 不同 Block 之间完全独立。
        
    - **Thread (线程)：** 最小的执行单元，负责处理具体的细分数据。
        
- **硬件物理层 (GPU 视角)：** * **Host -> Device 发射：** CPU 将任务以 `<<<Grid, Block>>>` 的形式下发给 GPU。
    
    - **SM (流多处理器) 接管：** 硬件会将 Block 作为一个不可分割的整体，分配给特定的 SM 执行。**核心机制：透明可扩展性 (Transparent Scalability)。** 无论你的显卡有几个 SM，硬件都会自动按需调度 Block，这保证了同一套代码可以在不同级别的显卡上无缝运行。
        

### 2. 微观调度机制：Warp 与指令执行

当 Block 被分配到 SM 后，微观层面的调度就开始了。

- **什么是 Warp？**
    
    Warp 是 SM 内部真正在硬件层面调度的基本单位。SM 会将 Block 里的线程按 **32 个一组** 强行打包成一个 Warp。这 32 个线程以 SIMT (单指令多线程) 的方式，同时执行同一条指令。
    
- **⚠️ 避坑指南：不可硬编码的 "32"：**
    
    尽管目前所有 NVIDIA GPU 的 Warp 大小均为 32，但**绝对不要在代码逻辑中硬编码 32**（如 `threadIdx.x % 32 == 0`）。NVIDIA 保留了未来架构升级更改此数字的权利。
    
    - **正确做法：** 在 Kernel 中使用内置变量 `warpSize`，在 Host 端使用 `cudaGetDeviceProperties` 动态获取。
        
- **Warp Divergence (分支发散)：**
    
    如果一个 Warp 内的 32 个线程遇到 `if-else` 分支，且产生分歧，GPU 无法同时执行两个分支。它必须串行处理：先执行 `if` 路径（挂起其余线程），再执行 `else` 路径。这会导致计算资源闲置，性能大幅下降。
    
    - **优化原则：** 尽量保证同一个 Warp 内的线程执行路径一致，避免条件分支直接依赖于 `threadIdx.x`。
        

### 3. Shared Memory (共享内存) 深度剖析

正如前文所述，Shared Memory 是为 Block 内部协作量身定制的硬件资源。

- **层级位置：** 位于 SM 内部，物理上非常靠近计算单元。类似于 CPU 的 L1 Cache，但由开发者**完全手动管理**。速度比 Global Memory (HBM/显存) 快上百倍。
    
- **核心作用：** 作为 Block 内线程通信的高速桥梁，也是缓存 Global Memory 中重复读取数据的“中转站”。
    
- **生命周期：** 与 Thread Block 强绑定。Block 启动时分配，Block 结束时立刻销毁。
    
- **声明方式：** `__shared__ float s_data[256];`
    

### 4. Bank Conflict (存储体冲突) 与规避

使用 Shared Memory 时，必须警惕底层的物理冲突。

- **什么是 Bank？**
    
    Shared Memory 在物理上被划分为 32 个大小相等的内存块，称为 Banks。其巧妙的设计目的是：允许一个 Warp 的 32 个线程同时访问时，如果每个线程恰好命中不同的 Bank，就能实现极高的并发内存带宽。
    
- **什么是 Conflict？**
    
    当一个 Warp 内的多个线程尝试访问**同一个 Bank 中的不同地址**时，硬件无法并行处理，只能将请求序列化（排队等待）。这就是 Bank Conflict，会导致原本极快的 Shared Memory 访存延迟成倍增加。
    
- **规避策略 (Padding 技术)：**
    
    最常见的冲突场景是按列读取二维数组。通过在数组的每一行末尾增加冗余的填充位 (Padding)，可以人为地错开不同行的数据在 Bank 中的对齐方式，从而打破冲突。
    
    - **冲突定义：** `__shared__ float tile[32][32];`
        
    - **解决思路：** `__shared__ float tile[32][33];` （Padding 技术，用极少的内存空间换取数倍的性能提升）。
