# 03 日志系统（app_logging）
本章解释日志如何被初始化、如何按“业务域（task_scope）”路由到不同文件、以及为什么在多进程环境下仍能安全写同一个日志文件。

- [app/application.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/app/application.py#L40-L57)

调用的是：
- `setup_logging(self.config.logging)`，实现位于 [app_logging/__init__.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/app_logging/__init__.py#L52-L167)

## 2. 路由规则核心：task_scope 优先，其次按模块路径
日志系统的设计意图写在模块 docstring 中：
- [app_logging/__init__.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/app_logging/__init__.py#L1-L22)

核心规则（非常重要）：
1. **优先看 `logger.contextualize(task_scope=...)` 注入的上下文**
2. 若没有上下文，则按 `record['name']`（模块路径）匹配

实现位置（filter 的逻辑）：
- [app_logging/__init__.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/app_logging/__init__.py#L106-L134)

规则展开：
- `task_scope='image_cell'` 或模块路径包含 `image_cell_processing` → 路由到 `image_cell` 日志文件
- 其他 `app/task_core/db/shared/config` 等通用模块 → 路由到 `app` 日志文件

## 3. 关键用法：在执行器中注入 task_scope
Image_cell 的执行器在执行每个任务时注入 `task_scope='image_cell'`：
- [image_cell_processing/orchestration/executor.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/orchestration/executor.py#L57-L66)

含义：
- 在该 `with` 代码块内，哪怕调用到 `db/` 或 `shared/` 的 logger，也会被路由到 image_cell 日志文件
- 这让你能“按业务域”拆分日志文件，而不是按模块碎片化拆分

## 4. 文件输出与多进程安全
日志 handler 添加位置：
- [app_logging/__init__.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/app_logging/__init__.py#L136-L151)

关键参数解释：
- `enqueue=use_async`：启用异步队列写日志，提高吞吐并减少阻塞
- `delay=True`：延迟创建日志文件，避免产生空文件
- `rotation/retention/compression`：自动轮转、保留与压缩

模块内也说明了不再使用“多进程集中式日志监听器”的原因：
- [app_logging/__init__.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/app_logging/__init__.py#L160-L165)

其核心假设是：
- 启动脚本（或环境变量）让所有进程写到同一条日志文件路径
- loguru 内部会处理多进程文件锁，从而保证并发写入的安全性

## 5. 与优雅关闭的关系：信号日志
信号处理器本身会打日志并设置关闭事件：
- [app/signals.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/app/signals.py#L12-L27)

这意味着你在日志里能看到：
- 收到 SIGINT/SIGTERM → 开始优雅关闭 → 各服务 stop 的流程

## 6. 常见排错建议（基于代码结构）
当你排查问题时，建议按以下顺序看日志：
1. 任务执行问题（推理、写库、瓦片）优先看 image_cell 日志文件（因 task_scope）
2. 启动失败/健康检查失败优先看 app 日志文件（application.py/health_service.py）

对应原因：
- 业务执行器明确注入 task_scope，而启动与基础设施不注入。
