## 1. 为什么需要 PipelineService（应用级单例）
PipelineService 的目标是：
- 把组件生命周期（start/stop/run loop）从“任务执行逻辑”中剥离
- 让多个任务复用同一套组件与共享资源（共享内存池、设备分配等）
- 在应用退出时统一清理

## 3. PipelineService.start()：启动组件并运行 run loop
启动入口：
- [pipeline_service.py:start](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/services/pipeline_service.py#L137-L175)

启动做了四件事：
1. builder.build() 得到 PipelineContext
2. 依序 `await component.start()`
3. 为每个组件创建 `asyncio.create_task(component.run())`（run loop 常驻）
4. 启动监控器（StatusMonitorWatcher）

停止时反过来：
- 先停止监控器
- 给 reader 输入队列塞 `None` 作为全局停止信号
- 等待组件 run loop 收敛
- **按逆序 stop**（下游先停，上游后停），确保排水逻辑生效

位置：
- [pipeline_service.py:stop](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/services/pipeline_service.py#L177-L208)

## 4. Runner：调度单个任务的“瓦片投喂与等待”
Runner：
- [core/runner.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/runner.py#L51-L188)

你可以把 Runner 理解为：
- “把一个 Task 变成一串 TileCoord”
- “把 TileCoord 投喂进流水线入口（TileReader.input_queue）”
- “等待 DBWriter 给出任务完成事件”

### 4.1 生成瓦片坐标：generate_tile_grid()
Runner 通过 tiling 算法生成坐标网格：
- [core/algorithms/tiling.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/algorithms/tiling.py#L13-L92)

该函数会打开 slide 读取尺寸（并立刻关闭），生成覆盖整个图像的坐标列表（齐平策略）。

### 4.2 将 TileCoord 投喂到 TileReader
核心投喂逻辑：
- [runner.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/runner.py#L146-L160)

它会：
1. 遍历 `grid.coords`，依次 `await reader.input_queue.put(coord)`
2. 最后发送 `TaskEndSignal(task_id)`，标记该任务数据流结束

### 4.3 等待结果收敛：DBWriter.wait_for_task_completion
Runner 通过 DBWriter 的任务完成事件等待该任务全部写入完毕：
- [runner.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/runner.py#L164-L188)

DBWriter 完成事件机制在：
- [core/components/db_writer.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/db_writer.py#L135-L163)
- [core/components/db_writer.py:on_finalize](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/db_writer.py#L198-L217)

关键点：只有当 TaskEndSignal 被一路广播到 DBWriter，并完成 flush 后，DBWriter 才会 `completion_event.set()`。

## 5. 队列数据模型：各阶段如何“约定”输入输出
数据模型集中在：
- [core/models.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/models.py#L39-L196)

四个核心结构：
- `TileCoord`：Runner → TileReader
- `ReadResult`：TileReader → TileFilter
- `FilterResult`：TileFilter/Normalizer → TileInferencer
- `InferResult`：TileInferencer → DBWriter / PointMapWriter

### 5.1 TaskEndSignal：任务级结束，而不是全局结束
`TaskEndSignal(task_id)` 的意义：
- 表示“该 task_id 的数据流结束”
- 组件收到它后会进行该任务的 finalize（等待在途项排空、释放资源），然后继续处理其他任务

定义：
- [core/models.py:TaskEndSignal](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/models.py#L186-L196)

全局停止（整个 pipeline stop）用的是 `None` 作为哨兵：
- BasePipelineStage.run 中对 `None` 的处理见 09 章（基类详解）。
