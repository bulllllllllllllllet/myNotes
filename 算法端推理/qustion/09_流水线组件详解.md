# 09 流水线组件详解（BasePipelineStage 与各阶段）
本章解释流水线为何能做到“多任务并行、阶段解耦、可排水停止、错误隔离、背压控制”。建议你对照源码逐段阅读。

## 1. BasePipelineStage：所有组件的共同骨架
基类：
- [core/components/base.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/base.py#L67-L359)

### 1.1 三类输入：数据项 / TaskEndSignal / None
`run()` 循环里有三类输入：
1. `None`：全局停止信号（pipeline stop 时由 PipelineService 发出）
2. `TaskEndSignal(task_id)`：任务级结束信号（Runner 发出）
3. 正常数据项：例如 `TileCoord / ReadResult / FilterResult / InferResult`

对应逻辑：
- [base.py:run](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/base.py#L136-L177)

### 1.2 TaskContext：active_count + zero_event（排水关键）
每个 task_id 对应一个 TaskContext，用于追踪“在途处理项计数”：
- `inc()`：派发前 +1
- `dec()`：处理结束 -1，归零时 set zero_event

定义：
- [base.py:TaskContext](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/base.py#L29-L65)

排水时（finalize）：
- 等待 `ctx.zero_event.wait()`，确保该 task 的所有在途项都处理完
- 然后执行 `on_finalize` 清理资源

对应流程：
- [base.py:_handle_finalize](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/base.py#L323-L359)

### 1.3 Pop-First 策略：避免“新旧任务复用同一上下文”
`_handle_finalize` 的第一步是把 context 从 `_contexts` 立即 pop 出来，转移到 `_finalizing_contexts`：
- 这样同一个 task_id 如果再次出现（极端情况），不会复用旧 context
- 统计仍可见（finalizing）

位置：
- [base.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/base.py#L334-L357)

### 1.4 错误隔离：process_item 报错不会杀死整个组件
`_process_wrapper` 捕获异常并调用 `on_error`：
- [base.py:_process_wrapper](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/base.py#L310-L322)

这保证了：
- 单个瓦片失败不会导致整个 pipeline 崩溃

## 2. SharedMemoryPool：背压与零拷贝的根基
共享内存池：
- [core/shared_memory.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/shared_memory.py#L38-L260)

## 3. TileReader：多线程读取 + 写入共享内存
实现：
- [core/components/tile_reader.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/tile_reader.py#L52-L249)

### 3.1 背压点：线程槽位 + 共享内存块双重 acquire
TileReader 在派发处理前先获取资源：
- 先 semaphore（并发线程槽位）  
- 再 shm_pool.acquire（共享内存块）  
如果 shm acquire 失败，会回滚 semaphore（避免隐性死锁）。

位置：
- [tile_reader.py:acquire_resources](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/tile_reader.py#L113-L140)

### 3.2 延迟打开 slide 句柄（每任务一个 handle）
TileReader 的 SlideContext 持有 `handle` 与 `lock`：
- 第一次处理该任务的瓦片时才打开
- 使用任务级 lock 避免并发重复打开

位置：
- [tile_reader.py:SlideContext](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/tile_reader.py#L35-L50)
- [tile_reader.py:process_item 延迟打开](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/core/components/tile_reader.py#L159-L170)

它调用的是 `get_slide(path)`，可根据文件扩展名选择 OpenSlide 或 KFB：
- [utils/slide_loader.py](file:///home/zyh/NewMedLabel/medlabel_image_cell_inference/src/image_cell_processing/utils/slide_loader.py#L49-L103)

### 3.3 共享内存释放时机
TileReader 成功广播 ReadResult 后不会释放 shm（交由下游释放）：
- 成功：把 `shm_name/shm_idx` 传给下游
- 失败：finally 里释放 shm_idx

## 8. 组件间“背压链路”总结（非常重要）
背压是贯穿全系统的稳定性设计：
- SharedMemoryPool.acquire 无可用块会阻塞 → Reader 不再产生 ReadResult
- Reader/Filter/Normalizer 各自的 semaphore 控制本阶段最大在途量
- BasePipelineStage 的输入队列 maxsize 控制阶段间缓冲容量
- DBWriter 的异步写入也由 semaphore 控制并发

当你遇到“处理慢/堆积”时，建议从这些背压点定位瓶颈：
- 哪个 input_queue.qsize 很大？
- SharedMemoryPool.available_count 是否接近 0？
- 哪个阶段的 semaphore 长期耗尽？
